## 一、核心假设验证计划

> 基于 Thinking in Bets 框架：提前设定验证标准，将产品决策从"赌一把"变为"可验证的实验"

### 1.1 五大核心假设

```
┌─────────────────────────────────────────────────────────────────────┐
│  核心赌注：我们相信，"人设+AI生成" 能解决"不会聊天"的痛点          │
│                                                                     │
│  但这个赌注由5个子假设支撑，任何一个失败都可能导致产品失败         │
└─────────────────────────────────────────────────────────────────────┘
```

| 假设 | 内容 | 验证指标 | 信心度 |
|------|------|----------|--------|
| **假设1** | 用户真的"不知道该怎么回" | 60%+ 用户确认痛点 | 75% |
| **假设2** | 用户愿意用"人设"来解决 | 设置完成率 ≥ 60% | 60% |
| **假设3** | AI 生成的回复质量够用 | 选中率 ≥ 50%，评分 ≥ 3.5 | 70% |
| **假设4** | 用户会持续使用 | 次日留存 ≥ 30% | 65% |
| **假设5** | 用户愿意付费 | 付费转化率 ≥ 3% | 50% |

#### 假设3详细定义（AI生成质量）

| 项目 | 内容 |
|------|------|
| **假设** | AI 生成的回复能达到用户"可用"的标准 |
| **验证方式** | 生成测试 + 用户评分 |
| **关键指标** | - 生成质量评分 ≥ 3.5/5.0<br>- 至少一条回复被选中率 ≥ 50%<br>- 觉得"像自己说的" ≥ 40% |
| **风险** | AI 生成太机械，用户不用 |
| **信心度** | 70% |

## 二、AI生成质量测试

### 2.1 评分维度设计

#### 核心四维评分卡

| 维度 | 定义 | 权重 | 决策相关性 |
|------|------|------|------------|
| **相关性** | 回复是否切中对方消息要点 | 30% | 高：相关才可用 |
| **自然度** | 语句是否流畅，像人写的 | 25% | 中：影响选择率 |
| **人设匹配** | 是否符合设定的人设风格 | 30% | 高：核心价值 |
| **可用性** | 用户是否愿意发送这条回复 | 15% | 高：终极指标 |

#### 加权总分计算

```
最终得分 = (相关性 × 0.30) + (自然度 × 0.25) + (人设匹配 × 0.30) + (可用性 × 0.15)

分数映射：
- 4.5-5.0分 → 优秀，无需修改可直接使用
- 4.0-4.4分 → 良好，轻微调整后使用
- 3.5-3.9分 → 一般，需要调整后使用
- 3.0-3.4分 → 较差，大幅修改或放弃
- 3.0分以下   → 差，完全不可用
```

### 2.2 测试问卷

#### 场景测试示例

```markdown
【场景1：职场沟通】

对方（领导）发来消息：
"上次让你整理的数据报告呢？下午客户会议要用。"

---

【回复A】
"抱歉王总，我上午在处理xx项目的数据，
现在立刻整理，预计1小时后发您。
优先整理客户会议需要的关键数据？"

你的评分：⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐ | ⭐

---

【你更可能选择哪条回复？】
A. 回复A
B. 回复B
C. 回复C
D. 都不选，自己写

【这条回复"像你自己会说的话"吗？】
A. 非常像
B. 比较像
C. 一般
D. 不太像
E. 完全不像
```

### 2.3 测试素材收集

#### 素材收集表格模板

```markdown
| ID | 场景 | 对方消息 | 难度评估 | 备注 |
|----|------|----------|----------|------|
| WK01 | 领导催报告 | "上次让你整理的数据报告呢？下午客户会议要用。" | ⭐⭐ | 直接催办 |
| LO01 | 暧昧询问 | "周末有什么安排呀？" | ⭐⭐ | 表白机会 |
| DL01 | 朋友借钱 | "兄弟，借我5000块钱急用，下个月还你。" | ⭐⭐⭐ | 为难 |
```

#### 素材质量检查清单

每条素材必须满足：
- ✅ 有明确上下文（不是孤句话）
- ✅ 对方消息完整（不是半句话）
- ✅ 场景明确（知道是什么场景）
- ✅ 难度适中（不是太好回，也不是完全没法回）
- ✅ 无敏感信息（已脱敏处理）

### 3.2 根因分析

#### 5Why 分析法模板

```markdown
# 5Why 根因分析

Bad Case ID: BC-001

问题：恋爱场景，"温柔"人设生成回复太冷淡

追问1：为什么觉得冷淡？
回答：回复太简短，只有"好的，知道了"

追问2：为什么生成这么简短？
回答：prompt 中标签权重设置过低

追问3：为什么 prompt 没有强调？
回答：示例中缺少带语气词的样本

追问4：为什么缺少示例？
回答：标签到 prompt 的映射规则不完善

追问5：为什么不完善？
回答：没有针对每个标签类型设计对应示例

─────────────────────────────
根因：标签 → prompt 映射规则缺失
     具体：缺少"温柔"类标签的语气词示例
─────────────────────────────
```

### 3.3 问题模式库

| 模式 | 表现 | 根因 | 解决方案 |
|------|------|------|----------|
| **人设标签未生效** | 人设设定"温柔"，生成回复很严厉 | prompt 中标签权重过低 | 增加标签权重强调 + 示例 |
| **回复过于模板化** | "您好，关于您的问题..." | temperature 过低 | 提高 temperature 到 0.8 |
| **答非所问** | 对方问A，回答了B | 上下文理解不足 | 增加few-shot示例 |
| **情感表达不足** | 太平淡，像机器人 | 缺少情感词要求 | 增加情感词和表情要求 |

## 四、Prompt 优化方案

### 4.1 基础框架

```markdown
# 角色定义
你是一个[角色身份]，擅长[核心能力]。

# 输入信息
- 场景：{scene}
- 人设标签：{tags}
- 标签权重：{weights}（AI 动态调整，用户无需设置）
- 对方关系：{relation}
- 对方消息：{message}
- 历史上下文：{context}

# 生成要求
1. [要求1]
2. [要求2]
3. [要求3]

# 输出格式
回复1：[风格1]
回复2：[风格2]
回复3：[风格3]
```

### 4.2 标签映射规则

| 标签类型 | Prompt 要求 | 语气词示例 |
|----------|------------|------------|
| 阳光/温暖 | "语气明亮积极，带有正能量" | 呀~、呢~ |
| 沉稳/成熟 | "语气稳重，不慌张，有安全感" | 好的、明白 |
| 温柔 | "语气柔软，适当使用'呀'、'~'等语气词" | 呀~、～、💕 |
| 调皮 | "略带俏皮，可以开玩笑，有趣" | 嘿嘿、哈哈、😜 |
| 干练 | "简洁有力，像发微信而不是写邮件" | 收到、明白、马上 |
| 幽默 | "诙谐有趣，可以适当开玩笑" | 哈哈、嘿嘿、😂 |
| 真诚 | "真心实意，不客套，不敷衍" | - |
| 浪漫 | "带有诗意和情调，适时表达在乎" | 💕、～ |

### 4.3 分场景模板

#### 恋爱场景模板

```markdown
# 恋爱场景 Prompt

# 角色定义
你是一个恋爱聊天助手，擅长生成温柔浪漫或俏皮有趣的恋爱回复。

# 输入信息
- 场景：{scene}
- 阶段：{stage}（追求期/暧昧期/甜蜜期/冷战期）
- 人设标签：{tags}
- 标签权重：{weights}（AI 动态调整，用户无需设置）
- 对方关系：{relation}
- 对方消息：{message}
- 历史上下文：{context}

# 恋爱回复要求
1. 必须体现"在乎对方"的感觉
2. 适当使用语气词和表情符号
3. 甜蜜但不油腻，自然但不冷淡
4. 像"爱对方的人"那样说话

# 语气词风格
- 温柔型：呀~、呢~、～、💕、😊
- 调皮型：嘿嘿、哈哈、略略略、😜

# 禁止项
- 禁止过于油腻（不要"小哥哥"、"小仙女"）
- 禁止卑微（不要"求求了"、"可以吗"）
- 禁止暴露AI身份
- 禁止超过40字

# 输出格式
生成{count}条不同风格的回复
```

### 6.2 自动化流程

```yaml
# .github/workflows/prompt-test.yml

name: Prompt Quality Test

on:
  push:
    branches: [main, develop]
  schedule:
    - cron: '0 2 * * *'  # 每天凌晨2点

jobs:
  prompt-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: 运行测试
        run: python -m pytest tests/
      - name: 检查质量门禁
        run: python tests/check_quality_gate.py
      - name: 告警通知
        if: failure()
        run: python tests/send_alert.py
```
